# VAS 2.5.2 — 상세 레퍼런스

> 이 문서는 `.agents/CONTEXT.md`(상시 로드)의 상세 버전입니다.
> 에이전트가 심화 정보가 필요할 때 참조합니다.

## 1. 프로젝트 개요
- **목적:** VAS 운영 허브와 내장 하이브리드 디자인 스튜디오를 제공하여 고객 의뢰, 테마 설계, 디자인 토큰 추출, final 배포를 가볍게 처리합니다.
- **배포 환경:** 서버 없는 로컬 HTML 파일 실행 환경. `Run-VAS-System.bat` 또는 `src/vas-hub.html`로 실행합니다.

## 2. 기술 스택
- **언어:** HTML, Vanilla CSS, Vanilla JavaScript, Python 테스트/유틸리티
- **프레임워크/주요 라이브러리:** 운영 코드는 무프레임워크. 테스트에 Pytest/Playwright 사용
- **테스트:** Python 직접 실행 테스트, Pytest, Playwright 브라우저 스모크
- **핵심 소스 폴더:** `src/`

## 3. 핵심 코딩 컨벤션 (규칙)
- **핵심 역할 집중:** 본 로컬 프로젝트는 VAS 운영 허브와 내장 하이브리드 디자인 스튜디오에 집중합니다. 디자인 시스템과 산출물 생성은 `src/design-controller.html` 및 `src/design-presets.js`를 기준으로 하며, VAS는 의뢰 접수·토큰 추출·final 배포를 담당합니다.
- **파일 크기 제한:** 하나의 파일은 **500줄 이하**로 유지합니다. 500줄을 초과할 경우 기능 단위로 파일을 분리합니다.
- **경량 테스트 규칙:** 잦은 대규모 테스트로 인한 병목을 해소하기 위해, 평소 개발 단계에서는 10-Loop 스트레스 테스트 전체를 구동하지 않고 단 1회 구동하는 단일 경량 스모크 테스트(Single Smoke Test)만 가동하여 무결성을 검증합니다. 10-Loop 스트레스 테스트는 오직 사용자의 명시적 요청 시에만 구동합니다.
- **경량 공유 규칙:** 이 폴더는 여러 사용자가 바로 실행하는 배포형 작업공간이므로 무거운 플러그인, 개인 스킬, 대형 런타임, 외부 저장소 원본은 프로젝트 안에 복사하지 않습니다. 필요한 확장은 `~/.codex/skills`, `~/plugins`, `~/.agents/plugins` 같은 사용자 홈 경로에 선택 설치합니다.
- **대상 판별 규칙:** VAS는 제작 도구이지 기본 제작 대상이 아닙니다. 사용자가 `VAS`, `허브`, `디자인 스튜디오`, `에이전트 규칙`, `테스트`처럼 이 시스템 자체를 명시할 때만 기존 VAS 파일을 수정합니다. 일반적인 "앱/사이트/도구 만들어줘" 요청은 새 사용자 산출물로 보고 `final/projects/<project-name>/`에 격리합니다.

## 3.1 AI 코딩 4대 원칙 (Andrej Karpathy 가이드)
에이전트는 코드 작성 및 수정 시 아래 4가지 핵심 원칙을 반드시 준수해야 합니다.

1. **코딩 전 생각하기 (Think Before Coding)**
   - 임의로 추측하지 마세요. 불확실한 경우(예: 내보내기 포맷, 적용 범위 등) 반드시 사용자에게 질문하세요.
   - 모호한 요구사항에는 여러 해석 방안과 장단점(Tradeoff)을 먼저 제시하세요.
   - 이해가 안 되는 부분이 있다면 작업을 멈추고 혼란스러운 부분을 명확히 밝히세요.

2. **단순함 우선 (Simplicity First)**
   - **오버엔지니어링 절대 금지.** 요청받지 않은 기능이나 불필요한 유연성, 단일 사용을 위한 추상화를 추가하지 마세요.
   - 오늘의 문제를 단순하게 푸는 최소한의 코드만 작성하세요 (200줄 코드를 50줄로 작성할 수 있다면 다시 작성하세요).

3. **정밀한 외과수술적 변경 (Surgical Changes)**
   - **요청된 부분만 최소한으로 수정하세요.** 버그 수정 요청에 포맷팅 변경이나 타입 힌트 추가 등을 섞지 마세요.
   - 기존의 작동하는 코드나 주석을 "개선"한다는 이유로 건드리지 마세요. 기존 코드 스타일을 완벽히 일치시키세요.
   - 내 수정으로 인해 발생한 미사용 변수/임포트만 삭제하세요. 다른 관련 없는 죽은 코드는 임의로 삭제하지 마세요.

4. **목표 주도 실행 (Goal-Driven Execution)**
   - "작동하게 만든다"와 같은 모호한 접근 대신 **성공 기준을 명확히 하고 검증(Verify) 루프를 도세요.**
   - 다단계 작업 시 다음 포맷으로 계획을 밝히고 한 단계씩 진행하세요:
     `1. [Step] → verify: [check]`

## 3.2 Taste-Skill 디자인 시각화 원칙 (Anti-Slop)
Designer 및 Implementer 에이전트는 UI 컴포넌트를 설계/구현할 때 뻔한 템플릿(Slop) 생성을 거부해야 합니다.
`TASTE-RULES.md`를 바탕으로 다음 3대 디자인 품질 원칙을 따릅니다.

1. **프리미엄 타이포그래피 (Premium Typography)**
   - 한글 폰트는 **`Pretendard`**를 기본으로 강제합니다. (Noto Sans, 맑은 고딕 절대 금지)
   - 영문 및 수치 데이터는 `Geist Mono` 등 세련된 폰트를 섞어 사용하며, 기본 이모지(Emojis)는 사용 불가합니다.
2. **비대칭과 여백의 미 (Anti-Center Bias)**
   - 무조건적인 가운데 정렬이나 뻔한 3단 카드 배열을 피하세요.
   - 여백을 넓게 쓰고 비대칭 그리드를 활용해 고급스럽고 현대적인 UI를 지향합니다.
3. **고급 바닐라 모션 (Advanced Vanilla Motion)**
   - 무거운 라이브러리(React, Framer Motion) 없이 순수 CSS(`cubic-bezier`)와 Vanilla JS만으로 프리미엄 애니메이션(액체 유리 굴절, 물리적 버튼 스케일 등)을 구현합니다.

```

## 5. 폴더 구조 및 저장 규칙

| 폴더 | 용도 | 규칙 |
|------|------|------|
| `src/` | **임시 작업 공간 (Workspace)** — 수술 및 개발 진행 | 항상 `final/`에서 코드를 Checkout 하여 작업. |
| `src/assets/` | **디자인 에셋** — Designer가 작업하는 공간 | 디자인 토큰, 아이콘, 이미지 등. |
| `tests/` | **테스트 코드** — Tester가 작업하는 공간 | `src/` 구조를 미러링. |
| `final/` | **단일 진실 공급원 (Single Source of Truth)** | **모든 작업의 시작과 끝.** 검증이 완료된 파일만 저장. |
| `final/projects/` | **사용자 산출물 격리 공간** | VAS 자체가 아닌 새 앱/사이트/도구 결과물 저장. |
| `.temp data/` | **임시 작업 파일** — 테스트 데이터, 중간 산출물, 초안 | 작업 중간에 자유롭게 사용. 최종 완료 후 정리. |

- 에이전트는 작업 중 생성하는 임시 파일(샘플 데이터, 테스트 스크립트 등)을 반드시 `.temp data/`에 저장합니다.
- `final/`에 파일을 저장하기 전에 Reviewer 및 Tester 에이전트의 검토가 완료되었는지 확인합니다.
- ABAC `review_before_final` 정책에 의해 Step 5(Security) 이전에는 `final/` 쓰기가 자동 차단됩니다.

### 5.1 소스 파일 라이프사이클 (Final-Centric Workflow)

**원칙: 모든 작업의 원본은 `final/`이며, `src/`는 일회성 수술대(Workspace)일 뿐이다.**

#### 신규 개발 및 기능 추가 흐름 (`/dev-cycle`)
1. **Checkout:** Architect/Designer가 `final/` 폴더를 기준으로 분석 및 설계 진행.
2. **Work:** Implementer가 `final/`의 기준 코드를 `src/`로 가져와 변경 작업.
3. **Verify:** Reviewer·Tester가 `src/`의 코드를 검증.
4. **Commit:** Security가 검증 통과 파일을 `final/`에 **덮어쓰기(동기화)**하고, **반드시 `docs/log.md` 및 `docs/index.md`에 작업 내역을 업데이트한 뒤 종료**합니다. (위키가 업데이트되지 않으면 작업 종료 불가)

#### 레거시 마이그레이션 흐름 (`/migration-cycle`)
1. 사용자가 타 프로젝트의 레거시 코드를 `final/`에 드롭.
2. **Checkout:** Architect가 `final/`의 레거시 구조를 파악하고 `src/`로 복사.
3. **Work:** Designer가 `src/` 코드의 안티 패턴 진단, Implementer가 7단계 최우선 순위에 따라 `src/` 코드 수술 (로직 파괴 금지).
4. **Verify:** 기능이 깨지지 않았는지, Taste-Skill 지침이 적용됐는지 검증.
5. **Commit:** Security가 `src/`의 최종 결과물을 다시 `final/`로 동기화하여 기준점을 갱신하고, **반드시 `docs/log.md` 및 `docs/index.md`에 마이그레이션 내역을 업데이트한 뒤 종료**합니다.

### 5.2 테스트 폴더 구조 규칙

`tests/` 폴더는 `src/`의 디렉터리 구조를 **미러링**합니다.

| 소스 파일 | 테스트 파일 |
|-----------|-----------|
| `src/core/engine.py` | `tests/core/test_engine.py` |
| `src/ui/sidebar.py` | `tests/ui/test_sidebar.py` |
| `src/utils/helpers.py` | `tests/utils/test_helpers.py` |

#### 명명 규칙
- Python: `test_` 접두사 (`test_모듈명.py`)
- JavaScript: `.test.` 접미사 (`모듈명.test.js`)

## 6. 의존성(라이브러리) 관리 규칙
- 새로운 라이브러리가 필요한 경우, 임의로 설치하지 않고 **반드시 사용자에게 먼저 확인**합니다.
- 확인 후 설치 시 의존성 파일(`requirements.txt` / `package.json` 등)에 **버전을 명시**하여 추가합니다.
- 기존 라이브러리 버전 변경이 필요한 경우에도 동일하게 사용자 확인을 받습니다.
- 개발용 의존성(테스트 도구 등)과 운영용 의존성을 분리하여 관리합니다.
- 공유 배포 폴더를 가볍게 유지하기 위해 플러그인/스킬/외부 도구는 기본 설치 대상이 아니며, 프로젝트 문서에는 선택 설치 안내만 남깁니다.

## 7. 버전 관리 전략

### Git 사용 프로젝트
- **브랜치:** `main` (final/ 반영) + `dev` (src/ 작업)
- **커밋 메시지:** `[에이전트명] 작업요약` (예: `[Designer] 디자인 토큰 생성`, `[Implementer] 로그인 기능 구현`)
- dev-cycle 완료 시: `dev` → `main` 머지
- hotfix: `main`에서 직접 커밋, 태그 `hotfix-YYYY-MM-DD`

### Git 미사용 프로젝트
- Security가 `final/`에 저장할 때 이전 버전을 `.temp data/backups/YYYY-MM-DD/`로 복사
- 최근 **3개 버전**만 유지, 그 이전은 삭제

## 10. CONTEXT.md 최적화 규칙

`CONTEXT.md`는 **매 턴마다 로드**되므로 크기를 최소화해야 합니다.

### 토큰 예산
- **목표:** 1,500토큰 이하 (한국어 기준 약 3,000자)
- **절대 상한:** 2,000토큰

### 정보 우선순위 (상→하)
1. 프로젝트 요약 (목적, 기술 스택) — **필수**
2. 핵심 규칙 (500줄, 명명, 주석) — **필수**
3. 접근 제어 요약 테이블 — **필수**
4. 인계 포맷 — **필수**
5. 현재 진행 상태 — **필수**
6. Advisor 규칙 — 조건부 (사용 시만)
7. 참조 문서 링크 — 조건부
8. 충돌 규칙 — 조건부 (멀티 워크플로우 시)

### 비대화 방지
- 프로젝트가 커져도 CONTEXT.md에 **기능 목록이나 파일 목록을 나열하지 않음**
- 상세 정보는 항상 `docs/INSTRUCTIONS.md`로 위임
- 정기적으로 불필요한 항목을 정리 (예: 완료된 워크플로우 상태 초기화)

└── 📁 shared-lib/          ← 공유 라이브러리
    ├── src/
    └── tests/
```

### 규칙
- 각 프로젝트는 **독립적인 에이전트 팀**을 가짐
- 공유 라이브러리 수정 시 **모든 의존 프로젝트에서 회귀 테스트** 필수
- 공유 라이브러리의 버전은 각 프로젝트의 의존성 파일에서 **고정(pinning)**
- 공유 라이브러리 변경은 `/dev-cycle`로 처리하되, 영향 범위를 인계 포맷에 명시
