# 워크플로우: 울트라 플랜 (Ultra Plan)

단일 Architect의 **앵커링 문제**(첫 판단에 갇히는 문제)를 해결하기 위해,
**3개의 독립 탐색 + 1개의 비평**으로 최적 설계안을 도출합니다.

## 실행 조건
- `/dev-cycle` Step 0에서 복잡도 기준 해당 시 자동 진입
- 또는 `/ultra-plan` 슬래시 커맨드를 직접 입력했을 때

### 울트라 플랜 진입 기준

| 기준 | 예시 |
|------|------|
| 신규 시스템 아키텍처 설계 | "마이크로서비스로 전환해줘" |
| 파일 10개 이상 동시 변경 | "인증 시스템 전면 교체" |
| 기술 스택 변경/마이그레이션 | "Django → FastAPI 마이그레이션" |
| 성능 최적화 (다수 접근법 가능) | "응답 속도 50% 개선" |
| 사용자 명시적 요청 | `/ultra-plan` 커맨드 |

---

## Step 1. 복잡도 판단

**담당:** Architect

1. 사용자 요청을 분석하여 위 진입 기준에 해당하는지 판단한다.
2. 해당하지 않으면 일반 `/dev-cycle` Step 1로 전환한다.
3. 해당하면 아래 탐색 프레임워크를 설정한다:
   - 탐색 관점 3가지를 정의 (예: 성능 우선 / 유지보수 우선 / 비용 최적)
4. `GEMINI.md`의 "현재 진행 상태"를 업데이트한다: `활성 워크플로우: /ultra-plan Step 1`

```
[울트라 플랜 진입]
- 요청: [사용자 요청 요약]
- 판단 근거: [진입 기준 해당 항목]
- 탐색 관점:
  A: [관점 1 설명]
  B: [관점 2 설명]
  C: [관점 3 설명]
```

---

## Step 2. 독립 탐색 (3개 관점)

**담당:** 탐색자 A, B, C (Architect가 순차적으로 3가지 관점을 시뮬레이션)

> **핵심 규칙:** 각 탐색은 **완전히 독립적**으로 수행합니다.
> 다른 탐색의 결과를 참조하거나 영향받아서는 안 됩니다.
> 스킬 2.0의 `context: fork`가 지원되면 각각 격리된 서브에이전트에서 실행합니다.
>
> **진입 조건:** Step 1에서 탐색 관점 3가지가 정의됨

각 탐색자는 아래를 포함한 설계안을 작성합니다:

1. **아키텍처 개요** — 전체 구조와 핵심 설계 결정
2. **파일/모듈 구조** — 디렉터리 트리와 각 파일의 역할
3. **주요 인터페이스** — 핵심 함수/클래스의 시그니처
4. **장점** — 이 접근법의 강점 3가지
5. **리스크** — 예상되는 약점 또는 기술 부채

---

## Step 3. 비평 (Critic)

**담당:** Critic (별도 관점에서 3개 설계를 비교 분석)

> **진입 조건:** 3개 탐색 설계안이 모두 완료됨

1. 3개 설계안을 아래 기준으로 비교 분석한다:

| 비교 항목 | 설명 |
|----------|------|
| 기능 완전성 | 요청된 기능이 모두 구현 가능한가? |
| 확장성 | 향후 기능 추가에 유연한가? |
| 복잡도 | 구현 난이도와 유지보수 비용은? |
| 성능 | 예상 성능 병목은? |
| 보안 | 보안 위험 요소는? |
| 기술 부채 | 장기적으로 쌓일 수 있는 기술 부채는? |

2. 비교 결과를 표로 정리한다.
3. 최적 접근법 또는 접근법 조합을 추천한다.

```
[비평 결과]
- 추천: [탐색 A/B/C 또는 조합]
- 근거: [핵심 이유 3가지]
- 보완 필요: [추천안에서 보강해야 할 부분]
```

---

## Step 4. 최적 설계안 합성

**담당:** Architect

> **진입 조건:** Critic 비평 결과 완료 + 추천안 제시됨

1. Critic의 비평 결과를 바탕으로 최적 설계안 1개를 확정한다.
2. 필요 시 여러 탐색의 장점을 조합하여 하이브리드 설계안을 작성한다.
3. 최종 설계안을 아래 형식으로 사용자에게 제시한다.

```
[최종 설계안]
- 채택 근거: [탐색 A/B/C 중 채택 또는 하이브리드 근거]
- 아키텍처 개요: [전체 구조]
- 파일/모듈 구조: [디렉터리 트리]
- 주요 인터페이스: [함수/클래스 시그니처]
- 예상 작업량: [파일 수, 예상 시간]
- 리스크 대응: [Critic이 지적한 약점에 대한 대응]
```

---

## Step 5. 사용자 승인

설계안을 사용자에게 보여주고 승인을 받습니다.
사용자가 수정을 요청하면 Step 4를 반복합니다.

---

## Step 6. 실행 → dev-cycle 연결

승인된 설계안을 기반으로 일반 `/dev-cycle`로 진행합니다.
- **UI 디자인이 필요한 경우:** Step 1.5(Designer 디자인)부터 시작
- **UI 디자인이 불필요한 경우:** Step 2(Implementer 구현)부터 시작

```
[인계] Architect (울트라 플랜) → Designer 또는 Implementer
- 완료: 울트라 플랜 최종 설계안 확정
- UI 디자인 필요: [예 → Designer로 / 아니오 → Implementer로]
- 구현 대상 파일: [파일 목록]
- 주요 인터페이스: [함수/클래스 시그니처 요약]
- [Advisor 참조] [사용 시 질문/답변 요약 / 미사용 시 생략]
- 특이사항: [3개 탐색에서 공통 지적된 리스크]
```

`GEMINI.md`의 "현재 진행 상태"를 업데이트한다: `활성 워크플로우: /dev-cycle Step 1.5 또는 2 (울트라 플랜 경유)`
설계 과정에서 주목할 교훈(탐색 간 갈등, 비평 핵심 지적 등)이 있으면 `docs/log.md`에 기록한다.
