# 워크플로우: 의뢰서 → 에이전트 팀 셋팅 자동 생성

고객이 제출한 「SW 프로젝트 의뢰 신청서」(JSON, PDF, 또는 이미지)를 분석하고,
VAS 시스템의 뼈대를 활용하여 **`final/[프로젝트명]/`** 폴더 하위에
독립적인 에이전트 팀 셋팅 파일 10개와 프로젝트 스캐폴딩을 생성한다.

## 실행 조건
- 사용자가 의뢰서(특히 `vas-client-requirements.json`)를 대화에 첨부했을 때 (JSON을 100% 파싱 우선)
- 또는 PDF/이미지를 첨부하거나 `/setup-from-application` 슬래시 커맨드를 입력했을 때

---

## Step 0. 스캐폴딩 대상 폴더(Target Directory) 결정

VAS 시스템(루트)을 오염시키지 않기 위해, 새 프로젝트는 반드시 `final/` 하위 폴더에 독립적으로 격리하여 생성한다.
- 추출한 프로젝트 이름을 바탕으로 **`final/[프로젝트명]/`** (예: `final/autocari/`) 폴더를 생성하고,
- 이후의 모든 셋팅 파일과 소스 폴더는 해당 폴더를 최상위 루트로 간주하여 내부에 구축한다.

---

## Step 1. 의뢰서 내용 파악

첨부된 의뢰서에서 아래 항목을 추출한다.
항목이 비어 있거나 불명확하면 `(미입력 - 추후 보완 필요)` 로 표시한다.

| 추출 항목 | 신청서 항목 |
|---|---|
| 프로젝트 이름 | 2단계: 프로그램 이름 |
| 프로젝트 목적 | 2단계: 해결하려는 불편함 |
| 참고 서비스 | 2단계: 참고할 서비스 |
| 주요 기능 (감각) | 3단계: 보거나 듣거나 읽어야 할 것 |
| 데이터 보유 여부 | 3단계: 관련 데이터 보유 여부 |
| 배포 환경 | 4단계: 사용 환경 |
| 마감일 | 5단계: 희망 완성 날짜 |
| 예산 범위 | 5단계: 예산 |
| 기타 요청사항 | 5단계: 기타 내용 |

---

## Step 2. 기술 스택 및 컨벤션 결정

의뢰서에서 파악한 내용(기능 유형, 배포 환경, 예산)을 바탕으로
아래 항목을 **전문가 판단**으로 결정한다. 사용자에게 일일이 묻지 않는다.

- **언어**: 감각 기능이 영상/음성이면 Python 우선, 웹이면 TypeScript 추가 검토
- **프레임워크**: 기능 유형에 맞는 표준 라이브러리 선택 (예: OpenCV, FastAPI, PyTorch 등)
- **테스트 도구**: Python이면 Pytest, JS이면 Jest
- **테스트 커버리지**: 예산 규모가 클수록 엄격하게 (기본 80% 이상)
- **명명 규칙**: Python이면 snake_case, JS면 camelCase
- **주석 언어**: 한국어 (기본값)
- **소스 폴더**: `src/` (기본값)
- **보안 요구사항**: 기타 내용란에 보안 언급 시 반영, 영상/음성 포함 시 메모리 폐기 규칙 포함

---

## Step 2.5. 접근 제어 모델 판단 (신규)

의뢰서의 배포 환경, 데이터 유형, 팀 규모를 분석하여 접근 제어 모델을 결정한다.

### 판단 기준

| 프로젝트 특성 | 적용 모델 |
|-------------|----------|
| 단순 스크립트/유틸리티 (1~3명) | **RBAC only** |
| 멀티 배포 환경 (dev/staging/prod) | **ABAC 추가** → `prod_config_lock` 정책 포함 |
| 민감 데이터 처리 (의료/금융/개인정보) | **ABAC 강화** → `sensitive_data_guard` 강화 |
| 영상/음성 처리 (카메라/마이크) | **ABAC 추가** → 메모리 폐기 정책 추가 |
| 대부분의 일반 SW 프로젝트 | **하이브리드 (기본)** → RBAC + ABAC 정책 3~5개 |

### Advisor 전략 적용 판단

- Claude API를 사용하는 프로젝트 → `advisor-strategy.md` 활성화
- 그 외 → `advisor-strategy.md`를 가이드라인 문서로만 포함

---

## Step 3. 셋팅 파일 10개 생성

앞서 결정한 **`final/[프로젝트명]/`** 폴더 하위에 아래 파일들을 생성한다. 모두 Step 1~2.5에서 파악한 내용으로 채운다.
(예: `final/autocari/GEMINI.md`, `final/autocari/.agents/access-control.md` 등)

```markdown
- 명명 규칙: [결정된 명명 규칙]
- 주석 언어: 한국어
- 임시 파일 → `.temp data/` | 최종 파일 → `final/`
- 🤖 **자연어 워크플로우 매핑:** 사용자가 슬래시 커맨드를 치지 않고 자연어로 요구하더라도 맥락을 파악해 스스로 적합한 워크플로우 가동
- ⚡ **Auto-Orchestrator Protocol:** AI는 `[인계]` 폼을 출력하고 사용자의 입력을 기다리지 마십시오. 스스로 바통을 넘겨받아 다음 에이전트 페르소나를 장착하고 즉시 다음 Step을 연속 실행(Auto-chaining)하여, 시작부터 최종 코드 저장 및 로그 업데이트까지 단 한 번의 세션으로 논스톱 완주하십시오.
```

### 3-3. `.agents/access-control.md` — RBAC+ABAC 접근 제어

- Step 2.5에서 결정된 접근 제어 모델로 생성
- RBAC 역할 매트릭스 + 판단된 ABAC 정책 포함
- 프로젝트 특성에 맞는 정책만 선별 포함

### 3-4. `.agents/advisor-strategy.md` — Advisor 전략

- Claude 사용 시 → 완전한 Advisor 전략 문서
- 비사용 시 → 가이드라인 참조 문서로 축소

### 3-5. `.agents/skills/architect/SKILL.md`

- 역할: 결정된 기술 스택과 기능 유형을 바탕으로 설계 에이전트의 역할을 구체적으로 작성한다.
- 스킬 2.0 YAML: `name`, `description`, `context: default`, `allowed_tools`, `denied_tools`
- RBAC 권한 섹션 포함
- Auto-Routing: 신규 기능 요청 또는 파일 구조가 아직 없는 상태에서 호출되도록 설정한다.

### 3-6. `.agents/skills/implementer/SKILL.md`

- 스킬 2.0 YAML + RBAC 권한 (쓰기 경로 제한)
- Auto-Routing 파일 경로는 `src/` 기준, 코드 패턴은 결정된 프레임워크 기준으로 명시
- `allowed_write_paths`, `denied_write_paths` 프로젝트에 맞게 설정

### 3-7. `.agents/skills/reviewer/SKILL.md`

- 스킬 2.0 YAML + `context: fork` + RBAC 권한 (쓰기 불가)
- 집중 검토 사항은 프로젝트 특성에 맞게 작성

### 3-8. `.agents/skills/tester/SKILL.md`

- 스킬 2.0 YAML + `context: fork` + RBAC 권한
- 필수 테스트 케이스를 기능 유형에 맞게 구체적으로 작성
- 커버리지 목표를 명시

### 3-9. `.agents/skills/security/SKILL.md`

- 스킬 2.0 YAML + `context: fork` + RBAC 권한 (`final/` 쓰기)
- 프로젝트의 데이터 유형에 맞는 보안 체크리스트 구성
- ABAC 연동 (sensitive_data_guard 등)

### 3-10. `.agents/skills/designer/SKILL.md`

- 스킬 2.0 YAML: `name: designer`, `context: default`, `allowed_tools`, `denied_tools`
- 역할: Claude Design 기반 디자인 시스템 구축 및 UI 프로토타입 생성
- RBAC 권한: `src/assets/`, `.temp data/` 쓰기 + 브라우저(리서치/Figma)
- Auto-Routing: 디자인, UI, 프로토타입, Figma 관련 요청 시 호출되도록 설정
- `allowed_write_paths`, `denied_write_paths` 프로젝트에 맞게 설정

## Step 4. 스킬 2.0 리소스 디렉터리 생성

각 스킬 폴더에 필요한 하위 디렉터리를 생성한다:

| 스킬 | scripts/ | references/ | assets/ |
|------|:--------:|:-----------:|:-------:|
| architect | - | ✅ (아키텍처 패턴 문서) | - |
| designer | - | ✅ (Claude Design 활용 가이드, Figma 연동 패턴) | - |
| implementer | ✅ (before_tool.sh/.ps1, after_tool.sh/.ps1) | - | ✅ (파일 템플릿) |
| reviewer | - | ✅ (코드 리뷰 체크리스트) | - |
| tester | ✅ (테스트 러너 .sh/.ps1) | - | ✅ (테스트 fixture) |
| security | ✅ (보안 스캔 .sh/.ps1) | ✅ (보안 체크리스트) | - |

> 프로젝트의 개발 환경(OS)에 따라 `.sh`(Linux/Mac) 또는 `.ps1`(Windows) 스크립트를 활성화한다.

## Step 4.5. 프로젝트 디렉터리 스캐폴딩

`final/[프로젝트명]/` 하위에 프로젝트에 필요한 기본 디렉터리와 설정 파일을 생성한다:

| 디렉터리/파일 | 용도 | 포함 파일 |
|----------------|------|----------|
| `index.html` | 새 프로젝트 메인 진입점 | `index.html` (빈 HTML 템플릿 또는 홈 화면) |
| `src/` | 소스 코드 | `README.md` (소스 폴더 안내) |
| `src/assets/` | 디자인 에셋 (Designer 작업 공간) | `README.md` (에셋 폴더 안내) |
| `tests/` | 테스트 코드 | `README.md` (테스트 구조 안내) |
| `final/` | 최종 산출물 | `README.md` (해당 프로젝트의 최종 결과물 폴더) |
| `.gitignore` | 버전 관리 제외 | `.temp data/`, `__pycache__/`, `node_modules/` 등 |
| `docs/log.md` | 히스토리 및 교훈 축적 | 빈 템플릿 (태그 목록 포함) |
| 의존성 파일 | 라이브러리 관리 | `requirements.txt`(Python) 또는 `package.json`(Node.js) |


## Step 6. 생성 파일 정합성 검증

생성된 파일들 간의 상호 참조가 일치하는지 자동 검증한다.

1. **기술 스택 일치:** `GEMINI.md`의 기술 스택 ↔ `INSTRUCTIONS.md`의 기술 스택 동일 여부
2. **도구 일치:** 각 `SKILL.md`의 `allowed_commands` ↔ 기술 스택 매칭 여부
   - 예: Python 프로젝트인데 `npm test`가 포함되어 있지 않은지
3. **RBAC 일치:** `access-control.md`의 RBAC 테이블 ↔ 각 `SKILL.md`의 RBAC 섹션 동일 여부
4. **경로 존재:** 모든 상호 참조 경로(예: `참조: .agents/xxx.md`)가 실제 존재하는지 확인
5. **불일치 발견 시** 자동 수정 후 사용자에게 보고
6. `GEMINI.md`의 "현재 진행 상태"를 업데이트하여 셋팅 완료를 기록한다.
