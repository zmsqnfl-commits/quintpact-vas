# 레거시 마이그레이션 워크플로우 (/migration-cycle)

이 워크플로우는 **기존 프로그램을 VAS 2.4 폴더에 흡수**하여,
VAS 인프라(에이전트 팀/워크플로우/디자인 시스템)를 그대로 활용하면서 개발을 이어가는 전체 사이클입니다.

## 사용 시나리오

| 케이스 | 설명 |
|--------|------|
| **A. 기존 프로그램 흡수** | 이미 동작하는 프로그램을 VAS 2.4 폴더에 넣어 에이전트 팀으로 관리/개선 |
| **B. 레거시 리디자인** | 기존 코드의 UI를 TASTE-RULES 기준으로 업그레이드 |
| **C. VAS 버전업** | VAS 2.3 이하 프로젝트를 2.4 인프라로 마이그레이션 |

> **진입 조건:** 사용자가 원본 프로젝트 경로를 알려주거나, 기존 코드를 `final/` 폴더에 직접 위치시킨 후 `/migration-cycle`을 호출합니다.

## Auto-Orchestrator 주의사항
- AI는 각 Step이 끝날 때마다 멈추지 않고, **즉시 다음 에이전트 페르소나를 장착하여 연속 실행(Auto-chain)**합니다.
- 사용자가 중단하라고 지시하기 전까지 Step -1부터 Step 6까지 논스톱 완주하십시오.

---

## Step -1. 프로젝트 인제스트 (Project Ingest) [Architect]

- **목적:** 원본 프로젝트를 안전하게 아카이빙하고 VAS 작업 폴더로 흡수합니다.
- **액션:**
  1. 사용자에게 원본 프로젝트 경로 확인 (이미 `final/`에 수동 배치된 경우 이 Step 생략).
  2. `python vas-migration-archive.py <경로>` 실행.
  3. `.temp data/프로젝트명_backup.zip` 생성 확인.
  4. `final/`에 프로젝트 파일 자동 배치 확인.
  5. 사용자에게 원본 폴더 삭제 여부 확인 → 승인 시 `--delete-source`로 재실행.
- **보관 정책:** 마이그레이션 백업은 `.temp data/`에 **최근 1개만** 유지 (새 백업 시 이전 `*_backup.zip` 자동 삭제).
- **종료 조건:** ZIP 아카이브가 생성되고 `final/`에 파일이 배치되었는가? (수동 배치 시: `final/`에 기존 코드가 있는가?)

---

## Step 0. 프로젝트 인식 및 VAS 인프라 세팅 [Architect]

- **목적:** 기존 프로그램의 정체를 파악하고, VAS 2.4의 GEMINI/INSTRUCTIONS를 해당 프로젝트에 맞게 재작성합니다.
- **액션:**
  1. **마이크로 백업:** `Run-VAS-Backup.bat` 실행하여 현재 VAS 인프라 스냅샷 생성.
  2. **`final/` 스캔:** 기존 프로그램의 파일 구조, 기술 스택(언어/프레임워크), 핵심 기능을 분석.
  3. **GEMINI.md 재작성:** `프로젝트 요약`을 기존 프로그램의 목적/스택으로 교체. VAS 인프라 섹션(핵심 규칙, 에이전트 인계 포맷, 접근 제어, Advisor 규칙 등)은 **절대 변경 금지**.
  4. **INSTRUCTIONS.md 재작성:** `프로젝트 개요`, `기술 스택` 섹션을 기존 프로그램에 맞게 교체. 섹션 3~8(코딩 컨벤션, 협업 규칙, 폴더 구조 등)은 **그대로 유지**.
  5. **docs/index.md 갱신:** 프로젝트 현황, 아키텍처 요약을 기존 프로그램 기준으로 업데이트.
- **보존해야 할 VAS 인프라 (절대 삭제 금지):**

  | 보존 대상 | 이유 |
  |-----------|------|
  | `.agents/` 전체 | 에이전트 팀, 워크플로우, 접근 제어 |
  | `docs/` 전체 | Wiki SSoT, 교훈 축적 |
  | `index.html` | 허브 페이지 |
  | `design-controller.html` + CSS | 디자인 시스템 코어 |
  | `client-application.html` + JS/CSS | 의뢰서 폼 |
  | `Run-VAS-System.bat` / `Run-VAS-Backup.bat` | 런처 |
  | `vas-backup.py` | 백업 시스템 |
  | `tests/` | 검증 인프라 |

- **종료 조건:** GEMINI.md/INSTRUCTIONS.md/docs/index.md가 기존 프로그램 맥락으로 갱신되었고, VAS 인프라는 온전한가?

## Step 1. Checkout 및 구조 분석 [Architect]

- **목적:** 기존 코드의 뼈대를 파악하고 수술대(`src/`)를 준비합니다.
- **액션:**
  1. `docs/index.md`를 읽고 프로젝트 최신 컨텍스트를 파악합니다.
  2. `final/` 폴더 내의 기존 코드 구조, 의존성, 핵심 로직을 스캔합니다.
  3. 기존 코드를 `src/` 폴더로 복사(Checkout)하여 작업 환경을 세팅합니다.
  4. 로직 파괴 방지를 위한 맵핑 문서(`src/assets/mapping.md`)를 작성합니다.
     - 원본 파일 경로 ↔ src/ 경로 대응표
     - 건드리면 안 되는 핵심 로직 목록
     - 500줄 초과 파일의 분리 계획
- **종료 조건:** 코드가 `src/`로 이동되었고 구조 분석이 완료되었는가?

---

## Step 2. Anti-Slop 디자인 진단 [Designer]

> **UI가 없는 프로그램(CLI, 백엔드 등)은 이 단계를 생략하고 Step 3으로 직행합니다.**

- **목적:** 기존 UI의 뻔한 안티 패턴(Slop)을 진단하고 리디자인 명세를 작성합니다.
- **액션:**
  1. `TASTE-RULES.md`를 바탕으로 `src/` 내 코드의 UI 취약점을 식별합니다.
     - 맑은 고딕/Arial → Pretendard 필요
     - 무조건 가운데 정렬 → 비대칭 레이아웃 필요
     - 뻔한 3단 카드 → 고급 컴포넌트 필요
  2. 사용자에게 `design-controller.html`에서 프리셋을 고르도록 안내합니다.
  3. `src/assets/`에 새로운 디자인 토큰을 정의합니다.
- **종료 조건:** 리디자인 명세서 및 토큰이 완료되었는가?

---

## Step 3. 외과수술 (Targeted Fix) [Implementer]

- **목적:** 기존 로직을 유지하며 타겟팅 수술을 진행합니다 (**완전 재작성 금지**).
- **액션:** 리스크가 적고 효과가 큰 순서대로 수정합니다.
  1. **500줄 규칙 적용:** 초과 파일을 모듈 분리합니다.
  2. **폰트 교체:** Pretendard 기본, 숫자에 Geist Mono (UI 있는 경우).
  3. **색상 정리:** 단일 악센트 + Off-Black 배경 적용 (UI 있는 경우).
  4. **모션 추가:** hover/active 마이크로 인터랙션 (UI 있는 경우).
  5. **레이아웃 재배치:** 비대칭 그리드, 여백 최대화 (UI 있는 경우).
  6. **상태 보강:** 로딩 스켈레톤, 에러 핸들링, 빈 화면 처리.
  7. **폴리싱:** 코드 컨벤션(kebab-case, camelCase), 한국어 주석 정리.
- **종료 조건:** 로직 파괴 없이 VAS 기준이 성공적으로 반영되었는가?

---

## Step 4. 무결성 검증 [Reviewer & Tester]

- **목적:** 퀄리티와 안정성을 동시 검증합니다.
- **액션:**
  1. **Reviewer:** 500줄 규칙, 접근 제어 준수, 디자인 명세 일치를 검토합니다.
  2. **Tester:** 기존 핵심 로직이 깨지지 않았는지 테스트합니다.
     - 기존 테스트가 있으면 실행, 없으면 핵심 기능 스모크 테스트 작성.
     - 경량 HTML/JS 프론트엔드는 Tester Bypass 적용 가능.
- **종료 조건:** 모든 테스트가 통과되고 리뷰가 승인되었는가?

---

## Step 5. Commit 및 동기화 [Security]

- **목적:** 수술이 완료된 코드를 기준점(`final/`)으로 돌려보냅니다.
- **액션:**
  1. 보안 취약점 스캔 (마이그레이션 중 추가된 외부 스크립트, API 키 노출 등).
  2. 검증 통과된 `src/`의 결과물을 `final/` 폴더로 병합/동기화(Commit).
  3. `src/` 폴더를 정리(수술대 클린업).
- **종료 조건:** `final/` 폴더가 최신 상태로 업데이트되고 `src/`가 비어있는가?

---

## Step 6. Wiki 업데이트 및 로그 [Security]

- **목적:** 마이그레이션 이력을 기록하고 프로젝트 위키를 최신화합니다.
- **액션:**
  1. `docs/log.md`에 마이그레이션 이력과 교훈을 기록합니다.
  2. `docs/index.md`를 최종 상태에 맞게 갱신합니다.
  3. `GEMINI.md`의 "현재 진행 상태"를 업데이트합니다.
  4. 로그 압축(Log Condensation)을 실행합니다.
- **종료 조건:** 모든 문서가 최신화되었는가?

---

## 사용자 퀵 가이드

```
방법 A (자동 아카이빙 — 권장):
  1. 채팅창에 "/migration-cycle" + 원본 프로젝트 경로 입력
  2. AI가 원본 → _backup.zip 아카이빙 → final/ 배치 → Step 0~6 자동 완주 → 끝

방법 B (수동 배치):
  1. 기존 프로그램 파일을 VAS 2.4의 final/ 폴더에 직접 복사
  2. 채팅창에 "/migration-cycle" 입력
  3. AI가 Step 0~6을 자동으로 완주 → 끝
```
