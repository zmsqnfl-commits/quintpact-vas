# 워크플로우: 개발 사이클 (Dev Cycle)

새로운 기능 구현 요청이 들어왔을 때, 에이전트 팀이 순서대로 협업하여
검증된 최종 결과물을 `final/` 폴더에 저장하기까지의 전체 흐름입니다.

## 실행 조건
- 사용자가 새로운 기능 구현을 요청했을 때
- 또는 `/dev-cycle` 슬래시 커맨드를 입력했을 때

## ⚡ Auto-Orchestrator 주의사항
- AI는 각 Step이 끝날 때마다 `[인계]` 폼을 출력하고 멈추는 것이 아니라, **즉시 다음 담당 에이전트의 페르소나를 장착하여 다음 Step을 연달아 실행(Auto-chain)** 해야 합니다. 사용자가 중단하라고 지시하기 전까지 0단계부터 6단계까지 단 한 번의 턴(Turn)으로 완주하십시오.

## 워크플로우 상태 추적

각 Step 시작 시 아래 형식으로 현재 상태를 기록한다:

```
[상태] /dev-cycle Step [N] — [에이전트명]
- 반복 횟수: [N]회차 (최대 3회)
- 이전 Step 결과: [통과 / 반려 / 최초 실행]
```

## Step 0. 복잡도 판단 및 마이크로 백업 (Micro-Commit)

**담당:** Architect (`.agents/skills/architect/SKILL.md` 참조)

1. **[필수] 마이크로 백업 실행:** 어떠한 코드 수정이나 설계를 시작하기 전, 무조건 터미널에서 `Run-VAS-Backup.bat` (또는 `python vas-backup.py`)을 실행하여 현재 상태의 안전망(Snapshot)을 `.vas_backups/`에 생성한다.
2. 요청의 복잡도를 판단한다.
2. **경량 모드(Lite) 기준에 해당하는 경우** → 아래 경량 흐름으로 진행한다.
   - 변경 파일 1~2개
   - 변경 줄 수 30줄 미만
   - 기존 인터페이스 변경 없음 (함수/클래스 시그니처 유지)
   - 사용자가 "간단히", "빠르게", "살짝" 등의 표현 사용
   - **경량 흐름:** Architect 생략 → Designer 생략 → Implementer → Reviewer(간소) → Security(final/ 이동만)
   - Tester/Security 보안 스캔은 변경 범위가 테스트·보안과 무관하면 생략 가능
   - UI 변경이 없는 경우 Designer 단계도 생략
3. **울트라 플랜 기준에 해당하는 경우** → `/ultra-plan` 워크플로우로 전환한다.
   - 신규 시스템 아키텍처 설계
   - 파일 10개 이상 동시 변경
   - 기술 스택 변경/마이그레이션
   - 성능 최적화 (다수 접근법 가능)
   - 사용자가 명시적으로 `/ultra-plan` 요청
4. 해당하지 않으면 아래 Step 1로 진행한다.
5. `.agents/access-control.md`의 RBAC/ABAC 정책을 확인한다.
6. `GEMINI.md`의 "현재 진행 상태" 섹션을 업데이트한다: `활성 워크플로우: /dev-cycle Step 0`

## Step 1. Architect — 설계

**담당 스킬:** `.agents/skills/architect/SKILL.md` (context: default)
**RBAC:** 읽기 전용 + 브라우저(리서치)

> **진입 조건:** Step 0에서 일반 복잡도로 판단됨 + 요청 사항이 명확함

1. **필수 작업:** 시작 전 무조건 `docs/index.md`를 읽어 프로젝트 현황과 아키텍처를 파악한다. 문서와 실제 코드가 불일치할 경우 기획 전에 문서를 먼저 패치(Patch)한다.
2. 요청 사항을 분석하여 전체 구조 설계안을 작성한다.
3. 신규 파일/모듈의 역할과 인터페이스를 정의한다.
4. 한 파일이 500줄을 초과할 것으로 예상되면 미리 파일 분리 계획을 제시한다.
4. 아키텍처 갈림길이 있으면 **Claude Advisor** 호출을 고려한다.
5. 설계 완료 후 아래 형식으로 Designer(또는 UI 변경 없으면 Implementer)에게 인계한다.

```
[인계] Architect → Designer
- 완료: 설계안 작성
- 구현 대상 파일: [파일 목록]
- 주요 인터페이스: [함수/클래스 시그니처 요약]
- UI 디자인 필요: [예 / 아니오 → 아니오이면 Designer 생략하고 Implementer로 직행]
- [Advisor 참조] [사용 시 질문/답변 요약 / 미사용 시 생략]
- 특이사항: [의존성, 주의사항 등]
```

## Step 1.5. Designer — 디자인

**담당 스킬:** `.agents/skills/designer/SKILL.md` (context: default)
**RBAC:** `src/assets/`, `.temp data/` 쓰기 | 브라우저(리서치/Figma)

> **진입 조건:** Architect 인계 완료 + UI 디자인이 필요한 요청 (UI 변경 없으면 이 단계 생략)
> 💡 **인간-AI 인터랙션:** UI 디자인이 필요한 경우, 사용자는 `Run-VAS-System.bat`을 실행하여 디자인 프리셋을 고른 뒤, 출력된 시스템 프롬프트를 Designer 에이전트에게 전달하면 완벽한 디자인 싱크로율을 낼 수 있다.

1. Architect 설계안과 **사용자가 제공한 [AI 에이전트 시스템 프롬프트]**를 기반으로 디자인 시스템을 구축한다 (디자인 토큰, 색상, 폰트, 간격 등).
2. **🔄 Architect 설계 정합성 확인:** 디자인이 Architect의 컴포넌트 구조·모듈 분리와 일치하는지 교차 검증한다.
   - 불일치 발견 시 Architect에게 피드백하여 설계안 조정을 요청한다 (Step 1 ↔ 1.5 반복, 최대 **2회**).
3. UI 컴포넌트 구조와 상태를 명세한다.
4. 프로토타입을 생성하여 사용자에게 확인받는다.
5. 시각 에셋(아이콘, 이미지 등)을 `src/assets/`에 저장한다.
6. 완료 후 아래 형식으로 Implementer에게 인계한다.

```
[인계] Designer → Implementer
- 완료: [디자인 작업 요약]
- Architect 설계 정합성: [확인 완료 / 조정 사항 요약]
- 디자인 토큰: [src/assets/design-tokens.json 등 파일 목록]
- 컴포넌트 명세: [컴포넌트 목록과 상태 요약]
- 에셋 파일: [src/assets/ 내 파일 목록]
- [Advisor 참조] [사용 시 질문/답변 요약 / 미사용 시 생략]
- 특이사항: [반응형 브레이크포인트, 접근성 요구사항 등]
```

## Step 2. Implementer — 구현

**담당 스킬:** `.agents/skills/implementer/SKILL.md` (context: default)
**RBAC:** `src/`, `.temp data/` 쓰기 | 터미널(빌드)
**ABAC 체크:** `large_file_alert`(400줄 초과 시 Reviewer 승인), `sensitive_data_guard`(민감 파일 차단)

> **진입 조건:** Designer 인계 완료(또는 Designer 생략 시 Architect 인계 완료) + 구현 대상 파일 목록 존재

1. Architect 설계안과 Designer 인계 내용(디자인 토큰, 컴포넌트 명세, 에셋)을 기반으로 코드를 작성한다.
2. `INSTRUCTIONS.md`의 코딩 컨벤션(명명 규칙, 주석, 500줄 제한)을 준수한다.
3. 작업 중 생성한 임시 파일은 `.temp data/`에 저장한다.
4. **접근 제어 범위 내에서만 작업**한다 (`denied_write_paths` 미접근).
5. **🔄 디자인 이슈 피드백:** 구현 중 디자인 명세의 기술적 문제(성능, 구현 불가능한 인터랙션 등)를 발견하면 Designer에게 피드백한다.
   - 디자인 수정이 필요한 경우: Step 1.5 ↔ 2 반복 (최대 **2회**). 초과 시 사용자 에스컬레이션.
6. 구현 완료 후 아래 형식으로 Reviewer에게 인계한다.

```
[인계] Implementer → Reviewer
- 완료: [구현한 기능 요약]
- 변경/생성 파일: [파일 목록]
- 디자인 명세 준수: [준수 / 조정 사항 있으면 요약]
- [Advisor 참조] [사용 시 질문/답변 요약 / 미사용 시 생략]
- 특이사항: [미구현 항목, 의존성 변경 등]
```

## Step 3. Reviewer — 코드 리뷰

**담당 스킬:** `.agents/skills/reviewer/SKILL.md` (context: fork — 격리 서브에이전트)
**RBAC:** 읽기 전용 + 터미널(린트만)

1. 구현된 코드의 품질, 가독성, 성능을 검토한다.
2. **접근 제어 준수 여부**를 확인한다 (Implementer가 RBAC/ABAC 범위 내 작업했는지).
3. **🔄 디자인 명세 일치 검토:** UI 관련 코드가 Designer의 디자인 토큰·컴포넌트 명세와 일치하는지 확인한다.
   - 디자인 불일치 발견 시 Implementer에게 수정 요청 (코드 문제) 또는 Designer에게 피드백 (디자인 명세 문제).
4. 문제 발견 시 Implementer에게 수정 요청하고 Step 2로 돌아간다.
   - ⚠️ **반복 제한:** Step 2↔3 반복 최대 **3회**. 초과 시 사용자에게 에스컬레이션.
5. 통과 시 아래 형식으로 Tester에게 인계한다.

```
[인계] Reviewer → Tester
- 완료: 코드 리뷰 통과
- 검토 파일: [파일 목록]
- 500줄 초과 파일: [있으면 파일명과 줄 수 / 없으면 "없음"]
- 접근 제어 준수: [준수 / 위반 시 상세]
- 디자인 명세 일치: [일치 / 불일치 시 상세]
- [Advisor 참조] [사용 시 질문/답변 요약 / 미사용 시 생략]
- 리뷰 결과 요약: [주요 수정 사항 또는 "이상 없음"]
- 특이사항: [테스트 시 주의할 엣지 케이스 등]
```

### 반려 인계 포맷

#### 코드 반려 (Step 3/4/5 → Step 2 역행)

Reviewer, Tester, Security가 **코드 문제**를 발견하여 Implementer에게 수정을 요청할 때 사용합니다.

```
[반려] [현재 에이전트] → Implementer
- 반려 사유: [구체적 문제 설명]
- 문제 위치: [파일명:줄번호 또는 함수명]
- 기대 수정 방향: [어떻게 고쳐야 하는지 구체적 가이드]
- 반복 횟수: [N]회차 / 최대 [M]회
- 이전 반려 이력: [있으면 이전 반려 사유 요약 / 없으면 "최초 반려"]
```

#### 디자인 반려 (Step 2/3 → Step 1.5 역행)

Implementer 또는 Reviewer가 **디자인 명세 문제**를 발견하여 Designer에게 수정을 요청할 때 사용합니다.

```
[디자인 반려] [현재 에이전트] → Designer
- 반려 사유: [디자인 명세의 구체적 문제 — 구현 불가, 성능 이슈, 접근성 미충족 등]
- 문제 항목: [디자인 토큰/컴포넌트명/에셋명]
- 기대 수정 방향: [어떻게 조정해야 하는지 가이드]
- 반복 횟수: [N]회차 / 최대 2회
- 이전 반려 이력: [있으면 이전 반려 사유 요약 / 없으면 "최초 반려"]
```

## Step 4. Tester — 테스트

**담당 스킬:** `.agents/skills/tester/SKILL.md` (context: fork — 격리 서브에이전트)
**RBAC:** `tests/`, `.temp data/` 쓰기 | 터미널(테스트만)

> **진입 조건:** Reviewer 통과 인계 완료 + 변경된 소스 파일이 `src/`에 존재

**[💡 Tester Bypass (경량 룰)]**
단위 테스트 프레임워크(Jest, Pytest 등)가 아예 구축되어 있지 않은 경량 HTML/JS 프론트엔드 프로젝트의 경우, 이 단계를 무시하고(Skip) 곧바로 **Security(Step 5)**로 인계하십시오. 무리하게 억지 테스트 코드를 짜려고 시도하지 마십시오.

1. 테스트 코드를 작성하고 실행하여 기능을 검증한다.
2. 테스트 실패 시 Implementer에게 수정 요청하고 Step 2로 돌아간다.
   - ⚠️ **반복 제한:** Step 2↔4 반복 최대 **3회**. 초과 시 사용자에게 에스컬레이션.
3. 통과 시 아래 형식으로 Security에게 인계한다.

```
[인계] Tester → Security
- 완료: 테스트 통과
- 커버리지: [달성 커버리지]%
- 테스트 결과: [통과한 주요 케이스 요약]
- [Advisor 참조] [사용 시 질문/답변 요약 / 미사용 시 생략]
- 특이사항: [보안 관련 주의사항 등]
```

## Step 5. Security — 보안 검토

**담당 스킬:** `.agents/skills/security/SKILL.md` (context: fork — 격리 서브에이전트)
**RBAC:** `final/` 쓰기 | 터미널(보안 스캔)
**ABAC 체크:** `review_before_final`(이 단계 이전에는 final/ 쓰기 차단), `prod_config_lock`(프로덕션 설정 보호)

> **진입 조건:** Tester 통과 인계 완료 + 모든 테스트 통과 + 커버리지 목표 달성

1. 보안 체크리스트를 실행하여 데이터 처리와 안전성을 검토한다.
2. ABAC `sensitive_data_guard` 정책 대상 파일의 보호 여부를 확인한다.
3. 문제 발견 시 Implementer에게 수정 요청하고 Step 2로 돌아간다.
   - ⚠️ **반복 제한:** Step 2↔5 반복 최대 **3회**. 초과 시 사용자에게 에스컬레이션.
4. 통과 시 최종 파일을 `final/`로 이동하고 사용자에게 보고한다.

```
[최종 보고] Security → 사용자(팀 리드)
- 완료: 전체 개발 사이클 통과
- 최종 파일 위치: final/[파일명]
- 보안 체크 결과: [통과 항목 요약 또는 "이상 없음"]
- 접근 제어 정책 준수: 확인
- docs/log.md 업데이트 필요 항목: [변경 이력 요약]
```

## Step 6. 지식베이스(Wiki) 업데이트 및 로그 압축 (Log Condensation)

**담당:** Security (`docs/` 쓰기 권한 보유)

마지막으로 프로젝트 지식베이스인 `docs/log.md`에 이번 작업 이력과 교훈을 추가한다.
주요 아키텍처나 기능이 변경된 경우 `docs/index.md`도 반드시 최신화한다.

**[중요] 로그 압축 (Log Condensation):** 
단순히 새 로그를 추가하기만 하면 컨텍스트 윈도우가 폭발합니다. 새 로그를 추가할 때, **오래되거나 중복되는 과거의 교훈들을 3~4줄 이내로 핵심만 요약 병합(Merge)**하여 `docs/log.md` 전체가 항상 가볍고 날카롭게 유지되도록 최적화(Pruning) 하십시오.

```markdown
## [날짜] [기능명]
- 변경 내용: [한 줄 요약]
- 교훈/특이사항: [발생한 버그, 해결책 등]
- 최종 파일: final/[파일명]
```

`GEMINI.md`의 "현재 진행 상태" 섹션을 업데이트하여 워크플로우 종료를 기록한다.

---

## 롤백 절차 (final/ 배포 후 문제 발견 시)

final/에 저장된 파일에서 사후 문제가 발견된 경우 아래 절차를 따릅니다.

1. **즉시 격리:** final/[파일명]을 `.temp data/rollback/[날짜]/`로 복사 (문제 파일 보존)
2. **복원:** 이전 정상 버전을 final/에 복원
   - `docs/log.md`에서 이전 정상 이력 확인
   - `.temp data/backups/` 또는 Git에서 이전 버전 가져오기
3. **분석:** 롤백된 파일로 `/hotfix` 시작
4. **기록:** `docs/log.md`에 `[ROLLBACK]` 태그로 이력 추가

```markdown
## [날짜] [ROLLBACK] [기능명]
- 롤백 사유: [한 줄 요약]
- 영향 파일: final/[파일명]
- 복원 버전: [날짜] 버전으로 복원
- 후속 조치: /hotfix [P레벨] 진행
```
