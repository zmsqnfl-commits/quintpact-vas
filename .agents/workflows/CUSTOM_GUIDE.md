# 커스텀 워크플로우 작성 가이드

프로젝트 고유의 워크플로우(예: `/deploy`, `/refactor`, `/data-pipeline`)를 만들 때 이 가이드를 참고합니다.

---

## 워크플로우 파일 구조

```yaml
---
description: 한 줄로 워크플로우 목적 설명
---
```

```markdown
# 워크플로우: [이름] (/슬래시-커맨드)

[목적 설명 1~2줄]

## 실행 조건
- [언제 이 워크플로우를 사용하는지]

---

## Step 1. [단계명]

**담당:** [에이전트명]

> **진입 조건:** [이 Step에 들어오기 위한 전제 조건]

1. [할 일]
2. [할 일]

---

## Step N. 완료 보고

`docs/log.md`에 이력을 기록한다.
`GEMINI.md`의 "현재 진행 상태"를 업데이트한다.
```

---

## 필수 포함 요소 체크리스트

| 요소 | 필수 | 설명 |
|------|:----:|------|
| YAML 프론트매터 `description` | ✅ | 슬래시 커맨드 목록에 표시됨 |
| 실행 조건 | ✅ | 언제 이 워크플로우를 쓰는지 |
| 각 Step별 담당 에이전트 | ✅ | SKILL.md 경로 포함 |
| 각 Step별 진입 조건 | ✅ | 중단 복구 시 판단 기준 |
| 각 Step별 RBAC/ABAC 명시 | ✅ | 권한 범위 명확화 |
| 인계 포맷 | ✅ | 표준 인계 포맷 사용 |
| 반복 제한 + 에스컬레이션 | ✅ | 무한 루프 방지 |
| docs/log.md 업데이트 | ✅ | 마지막 Step에서 |
| GEMINI.md 상태 업데이트 | ✅ | 시작/종료 시 |

---

## 네이밍 규칙

- 파일명: `kebab-case.md` (예: `dep-update.md`, `data-pipeline.md`)
- 슬래시 커맨드: `/kebab-case` (예: `/dep-update`)
- Step 번호: 0부터 시작 (사전 판단 필요 시) 또는 1부터 시작

---

## 등록 방법

1. `.agents/workflows/` 디렉터리에 `.md` 파일 생성
2. YAML 프론트매터에 `description` 포함
3. 에이전트 시스템이 자동으로 슬래시 커맨드로 인식

> 별도 설정 파일 수정 없이, 파일 생성만으로 등록됩니다.
