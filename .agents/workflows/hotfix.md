# 워크플로우: 긴급 수정 (Hotfix)

**정식 개발 사이클(`/dev-cycle`)과 달리 Architect 설계·Designer 디자인 단계를 생략**하고,
구현 → 리뷰 → 테스트 순서로 빠르게 처리합니다.
보안 검토는 변경 범위가 작으면 조건부로 생략할 수 있습니다.

## 실행 조건
- 서비스 중단이나 데이터 오류 등 즉각 수정이 필요한 버그가 발생했을 때
- 또는 `/hotfix` 슬래시 커맨드를 입력했을 때

## 워크플로우 상태 추적

각 Step 시작 시 아래 형식으로 현재 상태를 기록한다:

```
[상태] /hotfix Step [N] — [에이전트명]
- 긴급도: [P0 / P1 / P2]
- 반복 횟수: [N]회차 (최대 2회)
- 이전 Step 결과: [통과 / 반려 / 최초 실행]
```

> **⚠️ 주의:** Hotfix는 긴급 상황 전용입니다. 신규 기능 추가나 리팩토링에는 `/dev-cycle`을 사용합니다.

---

## 긴급도 레벨 분류

| 레벨 | 기준 | ABAC 권한 변경 |
|------|------|---------------|
| **P0** | 서비스 완전 중단, 데이터 유실 위험 | `hotfix_escalation` 적용: Implementer에게 `final/` 쓰기 + 자동 승인 임시 부여 |
| **P1** | 주요 기능 장애, 워크어라운드 가능 | RBAC 기본 권한 유지, Security 검토 필수 |
| **P2** | 경미한 버그, UX 이슈 | RBAC 기본 권한 유지, Security 검토 조건부 생략 가능 |

---

## Step 1. 버그 파악 및 긴급도 판정

1. 버그 증상과 발생 위치(파일명, 함수명)를 확인한다.
2. 긴급도 레벨(P0/P1/P2)을 판정한다.
3. 수정 범위를 최소화할 수 있는지 판단한다.
4. P0인 경우 ABAC `hotfix_escalation` 정책이 자동 적용됨을 확인한다.
5. `GEMINI.md`의 "현재 진행 상태"를 업데이트한다: `활성 워크플로우: /hotfix Step 1 (P레벨)`
6. 아래 형식으로 Implementer에게 즉시 인계한다.

```
[긴급 인계] → Implementer
- 버그 증상: [어떤 오류가 발생하는지]
- 발생 위치: [파일명, 함수명]
- 긴급도: [P0 / P1 / P2]
- ABAC 권한 변경: [P0이면 hotfix_escalation 적용 / 아니면 "기본 RBAC"]
- 예상 원인: [간단히]
- 수정 범위: [최소 수정 대상 파일]
```

---

## Step 2. Implementer — 긴급 수정

**담당 스킬:** `.agents/skills/implementer/SKILL.md`
**RBAC:** 기본 `src/`, `.temp data/` 쓰기
**ABAC (P0 시):** `hotfix_escalation` → `final/` 직접 쓰기 허용 + 자동 승인

> **진입 조건:** Step 1 인계 완료 + 버그 증상/위치/긴급도 확인됨

1. 버그 원인을 정확히 파악하고 최소 범위로 수정한다.
2. 수정 중 추가 리팩토링이나 기능 변경은 하지 않는다.
3. 완료 후 Reviewer에게 인계한다.

```
[인계] Implementer → Reviewer
- 완료: [수정 내용 한 줄 요약]
- 변경 파일: [파일 목록]
- 긴급도: [P0 / P1 / P2]
- [Advisor 참조] [사용 시 질문/답변 요약 / 미사용 시 생략]
- 특이사항: [연쇄 영향 가능성 등]
```

---

## Step 3. Reviewer — 빠른 코드 리뷰

**담당 스킬:** `.agents/skills/reviewer/SKILL.md` (context: fork)
**RBAC:** 읽기 전용 + 터미널(린트만)

> **진입 조건:** Implementer 수정 완료 인계 + 변경 파일 목록 확인됨

1. 수정된 코드만 집중 검토한다 (전체 리뷰 생략).
2. 수정으로 인한 **연쇄 영향(Side Effect)** 여부를 반드시 확인한다.
3. 문제 발견 시 Implementer에게 수정 요청하고 Step 2로 돌아간다.
   - ⚠️ **반복 제한:** Step 2↔3 반복 최대 **2회** (hotfix는 신속성 우선). 초과 시 사용자 에스컬레이션.
4. 완료 후 Tester에게 인계한다.

```
[인계] Reviewer → Tester
- 완료: 빠른 리뷰 통과
- 500줄 초과 파일: [있으면 파일명과 줄 수 / 없으면 "없음"]
- 연쇄 영향: [있음/없음 + 영향 범위]
- [Advisor 참조] [사용 시 질문/답변 요약 / 미사용 시 생략]
- 특이사항: [없음]
```

### 반려 인계 포맷 (Step 3/4 → Step 2 역행 시)

Reviewer 또는 Tester가 문제를 발견하여 Implementer에게 수정을 요청할 때 사용합니다.

```
[긴급 반려] [현재 에이전트] → Implementer
- 반려 사유: [구체적 문제 설명]
- 문제 위치: [파일명:줄번호 또는 함수명]
- 기대 수정 방향: [어떻게 고쳐야 하는지 구체적 가이드]
- 반복 횟수: [N]회차 / 최대 2회
- 이전 반려 이력: [있으면 이전 반려 사유 요약 / 없으면 "최초 반려"]
```

## Step 4. Tester — 핵심 케이스만 테스트

**담당 스킬:** `.agents/skills/tester/SKILL.md` (context: fork)
**RBAC:** `tests/` 쓰기 + 터미널(테스트만)

> **진입 조건:** Reviewer 통과 인계 완료 + 수정된 코드가 src/에 존재

1. 버그가 재현되지 않는지 핵심 케이스만 빠르게 검증한다.
2. 연쇄 영향이 있는 경우 해당 기능도 추가 검증한다.
3. 테스트 실패 시 Implementer에게 수정 요청하고 Step 2로 돌아간다.
   - ⚠️ **반복 제한:** Step 2↔4 반복 최대 **2회**. 초과 시 사용자 에스컬레이션.
4. **P0/P1:** Security에게 인계한다 (Step 5 진행).
5. **P2 (Security 생략 가능):** `final/` 쓰기 권한이 Tester에게 없으므로, 사용자에게 최종 파일 확인을 요청하고 **사용자가 직접 `final/`로 이동**한다.

```
[인계] Tester → Security
- 완료: 핵심 테스트 통과
- 버그 재현: 안 됨 (해결 확인)
- 연쇄 영향 검증: [통과 / 해당 없음]
- [Advisor 참조] [사용 시 질문/답변 요약 / 미사용 시 생략]
- 특이사항: [보안 관련 주의사항]
```

---

## Step 5. Security — 조건부 보안 검토 (P0/P1 필수, P2 조건부)

**담당 스킬:** `.agents/skills/security/SKILL.md` (context: fork)
**RBAC:** `final/` 쓰기

> **진입 조건:** Tester 통과 인계 완료 (P0/P1 필수, P2 조건부)

- **P0/P1:** 보안 체크리스트 실행 후 `final/`에 저장
- **P2:** 변경 범위가 보안과 무관하면 생략 가능

---

## Step 6. 완료 보고 및 docs/log.md 업데이트

**담당:** Security (P0/P1) / 사용자 직접 (P2 Security 생략 시)

```
[긴급 수정 완료] → 사용자(팀 리드)
- 수정 내용: [한 줄 요약]
- 긴급도: [P0 / P1 / P2]
- ABAC 임시 권한 사용: [사용함 / 사용 안 함]
- 최종 파일: final/[파일명]
- 잔여 과제: [못 고친 관련 이슈 / 없으면 '없음']
```

`docs/log.md`에 이력을 추가한다:
```markdown
## [날짜] [HOTFIX-P0/P1/P2] [버그명]
- 수정 내용: [한 줄 요약]
- ABAC 권한 변경: [hotfix_escalation 적용 여부]
- 최종 파일: final/[파일명]
```

> **ABAC 임시 권한 자동 만료:** P0에서 부여된 `hotfix_escalation` 확장 권한은 이 워크플로우 완료 시 자동으로 만료됩니다.

`GEMINI.md`의 "현재 진행 상태" 섹션을 업데이트하여 hotfix 워크플로우 종료를 기록한다.
주목할 만한 교훈(재발 방지 패턴, 근본 원인 등)이 있으면 `docs/log.md`에도 함께 기록한다.
